# 销售话术模拟案例

这份文档用于沉淀 LinearVision 当前阶段最常见的销售场景、客户异议和推荐回应。

核心目标不是“背台词”，而是把销售对话稳定拉回到三个问题：

- 你们现在的 Agent 是不是已经开始失控
- 谁在定义这些 Agent 的边界、权限和协作关系
- 如果规模继续扩大，谁来对错误决策和协同冲突负责

## 模块 1：客户说“我们已经用 Dify 先搭起来了”

### 客户原话

“我们内部已经用 Dify 搭了几个 Agent，先跑着看，暂时不需要外部方案。”

### 错误回应

“Dify 不行，还是要做专业架构。”

这种回答的问题是：

- 否定客户当前努力
- 把工具和架构混为一谈
- 很容易触发防御心理

### 推荐回应

“先跑起来是对的，很多团队都会先这么做。真正的问题通常不是第一个 Agent 能不能跑，而是跑到第十个、第十五个的时候，数据、权限、协作和责任边界是不是还清楚。我们更擅长处理的是这个阶段。”

### 继续推进的一句话

“我更想了解的是，你们现在这些 Agent 之间有没有统一的数据接入、权限控制和冲突处理机制？”

## 模块 2：客户说“我们团队自己能做，为什么要找你们”

### 客户原话

“我们内部工程师很强，这种事情自己也能搞。”

### 推荐回应

“这通常不是能力问题，而是视角问题。内部团队当然能把单个 Agent 做出来，但人和 AI 的边界、多个 Agent 的职责划分、权限设计、责任归属，往往没有人专门抽离出来系统设计。我们补的是这个空白。”

### 继续推进的一句话

“如果今天有两个 Agent 给出相互冲突的建议，或者一个 Agent 越权做了动作，内部现在是谁来定规则、谁来兜底？”

## 模块 3：客户还在蜜月期，不觉得自己有问题

### 客户原话

“我们现在效果挺好的，几个场景都已经跑起来了。”

### 推荐回应

“这很正常，早期通常都会有一个很好的蜜月期。问题一般不是出在前三个 Agent，而是出在规模开始增长之后。我们不会建议你们现在停下来重做，而是建议先做一次健康检查，看哪些地方未来最容易出问题。”

### 继续推进的一句话

“如果你们愿意，我们可以先不谈大项目，只做一次轻量的智能体健康检查，把风险点和优先级排出来。”

## 模块 4：客户已经开始混乱，但还没出事故

### 客户原话

“我们现在有十几个 Agent，不同团队都在搭，确实感觉有点乱，但还没严重到要重做。”

### 推荐回应

“这恰恰是最值得处理的时候。真正贵的不是重做，而是等到某个 Agent 的错误开始影响客户、财务或合规。我们可以先不大动，只先把职责、权限、交接点和冲突点梳理清楚。”

### 继续推进的一句话

“如果先做一个五天的 `AI Sprint`，我们可以很快帮你们看清楚：哪些 Agent 可以继续跑，哪些边界必须马上补。”

## 模块 5：客户已经出过事故

### 客户原话

“我们之前确实出现过 AI 输出错误、信息不同步、审批绕过，现在老板开始追问责任问题。”

### 推荐回应

“这就是我们最适合切入的场景。现在最重要的不是继续加新 Agent，而是先把现有系统的边界理清楚。我们会先帮你们把 `See / Think / Act` 三层梳理出来，再明确哪些决策必须回到人、哪些动作必须加验证和回滚。”

### 继续推进的一句话

“如果你们同意，我们下一步可以直接从一个出过问题的岗位或流程开始，五天内给出一版重构方案和原型。”

## 模块 6：客户问“你们和普通外包/咨询有什么不同”

### 客户原话

“这听起来像咨询，也像系统集成，你们到底和普通外包有什么区别？”

### 推荐回应

“外包通常解决的是‘按需求把东西做出来’，我们解决的是‘AI 在真实业务里怎么被安全、可控、可持续地接进去’。我们不是为了多堆几个 Agent，而是为了把信息、判断和动作重新连起来，并逐步沉淀成可复用的系统能力。”

### 简短版本

“外包卖工时，我们卖的是人机边界、协作结构和可持续落地。”

## 模块 7：客户问“为什么不能先继续凑合用”

### 客户原话

“我们现在先凑合用，等问题再大一点再说，不也可以吗？”

### 推荐回应

“可以，而且很多企业都会先这么做。问题在于，越晚梳理，未来要返工的范围越大。早期是修一个 Agent，后期往往是在修一串互相依赖的 Agent、权限和流程。我们不是让你们停下来，而是让你们在继续往前跑的时候，不把后面的重构成本越堆越高。”

### 继续推进的一句话

“我们可以先从最容易失控的一个流程开始，不用全盘重来。”

## 模块 8：客户问“你们第一步会怎么做”

### 客户原话

“如果我们要合作，你们第一步具体怎么做？”

### 推荐回应

“第一步不是做一堆方案汇报，而是进到你们真实岗位和流程里，看现在的 Agent 和人工到底怎么协作。我们通常会从一个 `AI Sprint` 开始：先跟岗、再切分人机边界、再搭一个真实原型，最后给出 ROI 和下一步部署建议。”

## 模块 9：面向技术负责人

### 关注点

- 现有系统是否还能扩
- 多个 Agent 之间如何协作
- 权限、治理、责任和可维护性

### 推荐说法

“我们不是来替代你们团队做 Agent，而是帮你们把已经出现或即将出现的复杂度控制住，避免以后每多一个 Agent，系统就更难管。”

## 模块 10：面向老板或业务负责人

### 关注点

- 会不会出事故
- 能不能省钱和提效
- 什么时候能看见结果

### 推荐说法

“我们不是先卖一个长期大项目，而是先用五天时间让你看到：哪些工作可以交给 AI，哪些必须留给人，继续做下去能省多少钱，哪些风险必须立刻处理。”

## 一页式销售主线

可以把大多数对话压缩成下面四句话：

1. “先跑起来是对的。”
2. “真正的问题通常出在规模扩张和跨 Agent 协作之后。”
3. “我们最擅长的不是做第一个 Agent，而是把已经开始失控的 Agent 系统重新理顺。”
4. “如果你愿意，我们先从一个五天的 `AI Sprint` 开始，不先卖大项目。”

## 使用建议

- 先判断客户处于蜜月期、混乱期、危机期还是重构期
- 不同阶段用不同问题切入，不要上来就讲同一套 PPT
- 先承认客户当前做法有其合理性，再指出其扩张后的结构性问题
- 每次对话都尽量落到一个明确下一步：健康检查、`AI Sprint`、单流程重构、案例复盘
