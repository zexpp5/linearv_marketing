# Agent 岗位与 KPI 设计

这份文档用于回答一个更具体的问题：

在 AI 时代，一个公司应该怎样定义一个 Agent 的岗位，以及该岗位的 KPI。

这里讨论的不是：

- 人的岗位 KPI
- Agent 编排平台选型
- 单纯的模型评测指标

这里讨论的是：

把一个 Agent 当成一个虚拟岗位，如何定义它的职责、边界、协作方式和绩效。

但这里要先加一个边界：

`Agent 岗位化` 更适合作为设计抽象，而不是默认事实。

换句话说：

- 不是所有 Agent 都天然等于一个岗位
- 不是所有 Agent 都值得单独设 KPI
- 很多场景里，真正应该被考核的是系统输出，而不是单个 Agent

## 先分清四个层级

| 层级 | 含义 | 典型对象 |
| --- | --- | --- |
| Tool | 工具层 | LLM / workflow / automation |
| Agent | 执行体 | 有目标、有流程、有限自主性 |
| Role | 职责层 | 有边界、有协作关系的设计抽象 |
| System | 系统层 | 真正承担业务结果的整体系统 |

这四层不能混在一起。

尤其要避免直接把：

`Agent = 岗位 = 责任主体`

这个等式当成事实。

## 核心判断

给 Agent 定 KPI，核心不是看“它多聪明”，而是看它是否稳定地创造业务结果。

最稳妥的方式，是把 KPI 分成四层：

- 效率
- 质量
- 财务
- 风险与安全

更进一步说：

- `Agent KPI` 更适合被视作诊断指标
- `System KPI` 才更接近真实业务考核对象

## 先定义岗位，再定义 KPI

在企业里，一个 Agent 应该先像 JD 一样被定义清楚。

至少要写清楚：

- 岗位名称
- 服务对象
- 职责范围
- 能做什么
- 不能做什么
- 协作模式
- 人工兜底机制

但只有在下面条件同时成立时，这种“岗位化定义”才真正有意义：

- 多 Agent 协同，不是单点工具
- 业务结果可测量
- 结果可归因
- 有清晰权限和交接边界

### 典型岗位名称

- 客服 Agent
- 销售跟进 Agent
- 数据分析 Agent
- 法务初审 Agent
- 合规审查 Agent
- 内部知识助手

## 一个 Agent 岗位应该怎么写

### 1. 岗位名称

例如：`客服回复 Agent`

### 2. 服务谁

例如：服务客服团队，承接标准化咨询

### 3. 职责范围

例如：

- 自动回复常见问题
- 自动识别高风险工单并转人工
- 自动生成初步解决建议

### 4. 权限边界

例如：

- 可以回复 FAQ
- 可以查询订单状态
- 不可以承诺赔付
- 不可以直接关闭争议工单

### 5. 协作模式

- 完全自主
- 人机协作
- 人工兜底

## KPI 设计的正确顺序

### 第一步：先定义业务目标

不要先问“Agent 能做什么”，而要先问“这个岗位最重要的业务结果是什么”。

例如：

- 客服 Agent：减少人工工单量
- 销售 Agent：提高线索转化率
- 运营 Agent：提升活动产出效率

### 第二步：再定义边界

Agent 更适合先进入：

- 流程明确
- 可以测量
- 风险低到中等

的场景。

如果边界不清，KPI 很容易失真，团队也容易为了指标掩盖异常。

### 第三步：再上 KPI

不要一开始堆很多指标。优先抓住：

- 1 到 3 个业务结果指标
- 少量质量指标
- 少量风险指标

## KPI 的真实对象

这是一个比“怎么设计 KPI”更关键的问题。

| KPI 层级 | 对象 | 是否推荐作为主考核对象 | 说明 |
| --- | --- | --- | --- |
| System KPI | 整个系统 | 是 | 最接近真实业务结果 |
| Process KPI | 某个流程环节 | 次优 | 有意义，但容易失真 |
| Agent KPI | 单个 Agent | 谨慎使用 | 更适合作为诊断指标 |

一个简单判断：

- 如果把 Agent 替换成函数或脚本，指标仍成立，那它更像系统 KPI，不是岗位 KPI
- 如果离开系统整体就无法归因，那单个 Agent KPI 的价值就很有限

## 四类 KPI 框架

### 1. 效率类

- 任务完成时间
- 自动化率
- 单位人力产出
- 平均响应时间
- 吞吐量

### 2. 质量类

- 一次解决率
- 准确率
- 可用回复率
- 人工返工率
- 漏判率

### 3. 财务类

- 节省成本
- 带来收入
- ROI
- 转化率提升
- GMV 提升

### 4. 风险与安全类

- 违规率
- 幻觉率
- 越权调用次数
- 人工介入次数
- 高风险误放行率

## 一个实用模板

每个 Agent 的 KPI 都可以用下面四句话先定下来：

1. 它要替谁解决什么问题。
2. 成功的定义是什么。
3. 允许的错误边界是什么。
4. 发生异常时谁来接管。

例如，一个 `客服回复 Agent` 可以这样写：

- 目标：减少人工客服负担
- 效率 KPI：首响时间降低 50%
- 质量 KPI：一次解决率达到 70% 以上
- 风险 KPI：高风险问题自动转人工，误放行率低于 1%

## 常见 Agent 岗位模板

### 客服 / 支持 Agent

岗位目标：承接尽可能多的简单咨询，同时不降低客户体验。

典型 KPI：

- 工单自动承接率
- 一次解决率
- 首次响应时间
- 转人工率
- 客户满意度
- 错答率

### 销售 / 线索跟进 Agent

岗位目标：在合规边界内，最大程度提升线索转化率。

典型 KPI：

- 自动跟进率
- 响应及时率
- 线索转化率
- 转人工率
- 异议识别准确率

### 运营 / 数据分析 Agent

岗位目标：自动完成日常数据监控、分析、报告和预警。

典型 KPI：

- 任务自动完成率
- 报告准时率
- 异常发现及时率
- 人工修正率
- 人工复核耗时

### 内部办公 / IT / HR 支持 Agent

岗位目标：降低重复咨询，提升内部员工效率。

典型 KPI：

- 自动解决率
- 平均响应时间
- 转工单率
- 员工满意度
- 人工支持时长节省

### 法务 / 合规 / 安全 Agent

岗位目标：在边界内做初筛，降低人工风险。

典型 KPI：

- 关键条款识别率
- 高风险事项标记率
- 漏标率
- 人工审核时长下降
- 安全违规率

## 设计原则

- 不要只看自动化率，高自动化可能只是把错误也自动化了
- 必须先有基准线，没有基准线，KPI 没有意义
- 不是所有 Agent 都要直接看营收
- 很多 Agent 应该先看效率和质量，再逐步升级到财务指标
- 对高风险流程，人工审核本身就是 KPI 体系的一部分
- 优先给系统输出定 KPI，再往下拆流程和 Agent 指标
- 警惕“指标优化”扭曲系统行为，例如为了追求自动化率而放大误判
- 不要把“数字员工”当工程事实，它更接近营销比喻

## 这是一个真实需求，还是伪需求

### 为什么这是真需求

- 企业已经开始把 Agent 纳入正式流程和治理体系
- 当 Agent 数量变多，必须有职责、边界和可观测指标，否则无法治理
- 合规、审计和责任归属都会逼企业定义“谁负责、谁审批、谁兜底”

### 什么情况下会变成伪需求

- 任务太简单，其实只需要脚本或函数，不需要岗位化
- 只是为了汇报和宣传，硬给 POC 写复杂岗位说明
- KPI 无法测量，或者与业务结果没有关系
- 人和多个 Agent 深度混合，根本无法归因
- 强认知、强创意任务，本身就不适合硬 KPI 化

## 一句话判断标准

如果企业要长期运维多个 Agent，并且这些 Agent 已经影响效率、质量、成本、合规或客户结果，那么 “Agent 系统治理 + 可归因 KPI” 就是真需求。

如果它只是一个简单自动化动作，或者暂时还停留在 demo 阶段，那就很容易变成伪需求。
