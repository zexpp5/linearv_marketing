# 外部研究

这个目录先统一放对定位直接有用的外部输入。当前最重要的不是泛泛研究市场，而是搞清楚四类目标客户分别想听什么。

## 四类目标客群的差异化信息需求

| 客群 | 他们最关心什么 | 推荐关键词 | 不要优先强调什么 |
| --- | --- | --- | --- |
| 银行与金融机构 | 合规、安全、可靠、可控 | 风控、监测、决策、可信、稳定 | 炫技、实验性、只讲提效 |
| 投资机构 | 信号、速度、判断、先手 | 市场信号、底层逻辑、机会、先于对手行动 | 复杂技术细节、泛泛自动化 |
| 科技公司 | 落地、集成、系统化能力 | 决策栈、工作流、嵌入业务、系统能力 | 只讲模型能力，不讲接入方式 |
| 创业团队 | 少人、高效、低成本、快执行 | 战斗力、效率、少花钱、多干事 | 架构、平台、抽象概念 |

## 当前研究结论

- 核心 tagline 不需要按行业拆分，品牌层统一使用 `See. Think. Act. With AI.`
- 真正需要按行业变化的是 tagline 下方的解释句，也就是“AI 到底替客户解决什么问题”。
- 银行和投资的进入门槛更高，但更适合高价值方案销售。
- 创业团队更容易进入，但对价格和交付周期更敏感。
- 当前阶段我们真正的对手，往往不是另一家供应商，而是客户内部“先凑合用 Dify / LangChain / 自己搭几个 Agent”的选择。

## 当前市场判断

截至 `2026-04-10`，对市场的内部判断是：

- 许多企业面对的真实选择不是“选谁做 AI 架构”，而是“要不要先自己凑合跑起来”。
- 这个“先凑合用”的选项在短期内看起来非常合理，因为成本低、启动快、汇报容易。
- 因此，市场不会先为“架构设计”买单，而会先为“事故后的重建、治理和清理”买单。

## 客户内部的四阶段演化

### 第一阶段：蜜月期

- 时间窗口：通常是部署后 `0-3 个月`
- 特征：Agent 数量不多，能跑简单任务，团队兴奋，老板觉得“我们已经在做 AI”
- 销售判断：这个阶段通常很难卖进“架构设计”

### 第二阶段：混乱期

- 时间窗口：通常是部署后 `3-9 个月`
- 特征：不同团队用不同工具搭了更多 Agent，没有统一数据标准、权限体系和协作逻辑
- 风险信号：Agent 之间开始出现冲突、重复、信息不同步

### 第三阶段：危机期

- 时间窗口：通常是部署后 `9-18 个月`
- 特征：错误决策、数据泄露、合规问题、客户投诉等开始造成真实损失
- 关键问题：老板开始追问“谁在管这些 AI，出了事谁负责”

### 第四阶段：重构期

- 时间窗口：通常在危机暴露后
- 特征：企业不得不重新梳理 Agent 的职责、权限、协作关系和治理机制
- 销售判断：这是最现实、也最容易成交的切入窗口

## 对 LinearVision 的意义

- 不要把主要精力放在说服蜜月期客户“事前设计有多重要”
- 优先寻找已经进入混乱期、危机期、重构期的企业
- 先解决“已经出问题”的现实痛点，再延伸到“事前设计”和长期架构
- 你们卖的不是一个更炫的 Agent，而是把失控的 Agent 系统重新理顺

## 当前最值得优先寻找的客户信号

- 已经部署了多个 Agent，但没有统一负责人
- 不同部门用不同工具各自搭建 Agent
- 出现过 AI 输出冲突、审批绕过、信息不同步、权限越界
- 内部开始出现“治理”“风险”“审计”“责任归属”这些讨论
- 技术负责人已经意识到“现在能跑，但扩不动，也不敢放大”

## 对标 Palantir 后对我们真正有用的结论

| 对标观察 | 对 LinearVision 的意义 | 当前动作 |
| --- | --- | --- |
| 对方卖的不是技术名词，而是不可替代的位置 | 我们不能只讲模型和 AI，要讲客户为什么离不开我们 | 把对外表达从“用了什么 AI”转成“连接了什么业务链路” |
| 对方的三段式表达背后有产品支撑 | `See / Think / Act` 不能只停留在 slogan | 逐步把三段式映射到真实能力、模块或交付方法 |
| 对方更会定义问题，而不是定义自己 | 我们要描述“没有 LinearVision 时的问题长什么样” | 优先沉淀问题场景、前后对比、案例故事 |
| 对方用 Bootcamp 这类机制缩短销售周期 | 我们也需要一个极短周期的价值验证方式 | 设计 `AI Sprint` 或 `5 天 AI 诊断` |
| 对方是产品公司，不是卖人头的服务公司 | 我们不能走向传统软件外包 | 服务要服务于产品 adoption，而不是靠堆人收费 |
| 对方今天可以不写 AI，是因为市场早已认识它 | 我们现在仍需要 `With AI` 作为认知标签 | 先保留 `With AI`，等品牌势能形成后再考虑弱化 |

## 对标时要学什么，不要学什么

要学的：

- 从感知到判断到行动的结构化表达
- 卖结果、卖关键位置，而不是卖技术参数
- 用 Demo、案例和短周期验证替代长篇解释

当前不该硬学的：

- 过强的统治型语气
- 还没有支撑时就说自己是操作系统
- 还没有产品化时就模仿成熟平台公司的叙事高度

## 如何加速市场教育

- 做 `AI 事故案例库`，系统收集企业 Agent 部署失败、失控和重构案例
- 做免费的 `智能体健康检查`，先让客户意识到自己已经有治理问题
- 强绑定 `事故后重建` 场景，而不是只卖“事前规划”
- 去找技术社区、行业会议、编排工具生态里那些已经在抱怨 Agent 混乱的团队

## 当前最佳策略

- 保持小团队、低成本、高密度地做项目
- 不急着大规模品牌投放，先积累实战案例和失败后重建经验
- 每做一个重构项目，就沉淀一份方法论、一组风险模式和一套销售故事
- 当监管、保险、客户审计这些外部力量加强时，已经做过最多实战的人会更容易赢

## 后续收集建议

- 收集四类客户在官网、路演、会议里常用的关键词
